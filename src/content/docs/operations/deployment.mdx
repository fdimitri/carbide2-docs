> This page covers the deploy workflow once a cluster exists.

## deploy.rb

The primary deploy tool is `scripts/deploy.rb` in the meta repo (`carbide2`).
It self-updates the meta repo checkout, refreshes submodules, imports images,
and rolls the control plane + workspace deployments.

```bash
cd ~/repos/carbide2

# Full deploy (control plane + all workspace pods)
ruby scripts/deploy.rb

# Control plane only
ruby scripts/deploy.rb --roll-scope control

# Workspace pods only
ruby scripts/deploy.rb --roll-scope workspaces
```

### Long-running deploys

Run with `nohup` to avoid terminal hang and watch the logfile while the deploy runs:

```bash
nohup ruby scripts/deploy.rb > /tmp/deploy.log 2>&1 &
tail -n 40 /tmp/deploy.log
```

## Submodule pin discipline

Before every deploy, ensure all four submodule pins are current on meta `main`:

```bash
# Check current pin SHAs
git submodule status

# Bump a component (example: server)
cd carbide2-server && git checkout main && git pull
cd .. && git add carbide2-server && git commit -m "bump carbide2-server"
git push origin main
```

Stale pins build old images. Checklist: **server**, **control**, **client**, **worker** — all four must be current.

Pinned submodules answer which code the deploy builds. They do not explain an
old SPA still being served after a deploy if the new image was already built and
imported successfully.

## Deploy pulls meta `main`

`deploy.rb` does `git pull --ff-only` on the **current branch** of the meta repo
checkout on the deploy target (typically `main`). Changes on `dev` only ship after
merging to `main`.

## Same-tag rebuilds still need a rollout

`deploy.rb` force-restarts the control-plane deployments and existing workspace
deployments after re-importing images. That matters because a Helm upgrade is a
no-op when only the image contents changed under the same tag.

If an old SPA is still being served after deploy, look at rollout state first:
- old pods may still be serving traffic
- the relevant deployment may not have restarted cleanly
- stale shell sidecar pods may still be running old code until deleted and respawned

## Fast iteration (code overlay)

For rapid dev cycles without a full image rebuild:

```bash
# Overlay changed files into the running pod
./scripts/dev-native.sh
```

Full rebuild is only needed for gem or npm dep changes.
