## Deploy built old code despite new commits

**Cause:** the meta repo pins old component SHAs. `deploy.rb` builds from the
pinned SHAs, not from component branch tips.

**Fix:**
```bash
cd ~/repos/carbide2
git submodule update --init --recursive
git submodule status   # verify all four pins are current
```

Checklist before redeploy: **server, control, client, worker** pins all current
on meta `main` — the deploy pulls meta `main`, not `dev`.

## Old SPA served after deploy

This is usually a rollout or running-pod problem, not a submodule-pin problem by
itself. A stale client pin explains why an old SPA image got built; it does not
explain why an already-built new image is not what clients are seeing.

Check these first:

- Did the relevant deployment actually roll after the image import?
- Are old control-plane or workspace pods still serving traffic?
- If this was a same-tag rebuild, did the forced rollout complete?
- Are stale shell sidecar pods still hanging around after the worker restart?

## 404 on an authenticated endpoint (expected 401)

The API base controller now returns **401 "Token does not match a known user"**
when a token's `sub` has no matching user in this pod's database. If you still
see a 404 on an authed endpoint, it is almost always a scoped resource lookup
(`current_user.projects.find(...)`) — meaning the resource doesn't exist *or*
the user isn't a member. Check membership before assuming a routing bug.

## Browser shows stale behaviour that curl can't reproduce

The SPA may be replaying stale `localStorage` tokens/state. Reproduce in a real
browser session (or Playwright with fresh storage) before concluding the server
is at fault.

## k3d networking on WSL

If the cluster is unreachable after a Windows sleep/resume cycle, restart the
k3d cluster: `k3d cluster stop carbide && k3d cluster start carbide`.
