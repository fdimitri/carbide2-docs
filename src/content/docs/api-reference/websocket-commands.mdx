The protocol architecture (wire format, versioning, handler routing) is described
in [Architecture → WebSocket Protocol](/architecture/websockets/). This page is
the per-command reference.

Security levels (`N0..N3`) follow [Security Model](/architecture/security-model/).

## Message envelope

```json
{ "cs": "<commandSet>", "cmd": "<command>", "payload": {} }
```

Current worker messages use only these three top-level fields. A request `ref`
field is not implemented on the wire today.

## `system`

| Command | Direction | Security | Payload |
|---------|-----------|----------|---------|
| `connected` | S→C | N1 | `{ user_id, project_id, protocol, min_client, token_exp }` |
| `error` | S→C | N1 | `{ message }` |
| `pong` | S→C | N1 | `{ t }` |
| `reauth_ok` | S→C | N1 | `{ exp }` |
| `reauth_failed` | S→C | N1 | `{ message }` |
| `token_expired` | S→C | N1 | `{}` |

## `fs`

| Command | Direction | Security | Payload |
|---------|-----------|----------|---------|
| `tree` | C→S | N2 | Request payload ignored; reply is `fs/tree` with `{ tree }` |
| `read` | C→S | N2 | `{ path }` → reply `fs/content` with `{ path, content }` |
| `read_binary` | C→S | N2 | `{ path, offset?, length? }` → reply `fs/binary_chunk` with `{ path, offset, length, size, eof, data }` |
| `stat` | C→S | N2 | `{ path }` → reply `fs/stat` with stat fields for that entry |
| `open` | C→S | N2 | `{ path }` → reply `fs/opened` with `{ path, viewers }` |
| `close` | C→S | N2 | `{ path }` |
| `cursor` | C→S | N2 | `{ path, line, char }` → peer broadcast `fs/cursor` with `{ path, user_id, name, line, char }` |
| `write` | C→S | N2 | `{ path, changes: [{ change_type, change_data, start_line, start_char, end_line, end_char }] }` |
| `set_contents` | C→S | N2 | `{ path, content }` |
| `create_file` | C→S | N2 | `{ path, content?, mkdirp? }` → reply/broadcast `fs/created` |
| `create_dir` | C→S | N2 | `{ path }` → reply/broadcast `fs/created` |
| `rename` | C→S | N2 | `{ path, new_name }` → reply/broadcast `fs/renamed` |
| `delete` | C→S | N2 | `{ path }` |
| `import_git` | C→S | N2 | `{ git_url, git_ref? }` → `fs/import_started`, then `fs/import_done` or `fs/error` |
| `change` | S→C | N2 | Broadcast: `{ path, change_type, change_data, start_line, start_char, end_line, end_char, revision, user_id }` |
| `set_contents` | S→C | N2 | Broadcast: `{ path, content, revision, user_id }` |
| `created` | S→C | N2 | Broadcast/reply: `{ path, type, id, user_id? }` |
| `renamed` | S→C | N2 | Broadcast/reply: `{ old_path, new_path, id, user_id? }` |
| `deleted` | S→C | N2 | Broadcast/reply: `{ path, user_id? }` |
| `changed` | S→C | N2 | Watcher broadcast for on-disk mutations |
| `error` | S→C | N2 | `{ message }` or `{ path, error }` |

## Fanout semantics

| Stream | Fanout | Durability | Recovery |
|--------|--------|------------|----------|
| `fs/change` | broadcast to other clients viewing the same open file | backed by DB changes | re-read file (`fs/read`) |
| `chat/message` | broadcast to room members | durable in `chat_messages` | request history |
| `term/output` | broadcast to subscribers of terminal id | ephemeral stream | reopen terminal; no guaranteed replay |
| `agent/done` | final response to conversation listeners | final assistant message persisted | reload history |

## Concrete examples

### Example: `fs/tree`

Client → server:

```json
{ "cs": "fs", "cmd": "tree", "payload": {} }
```

Server → client:

```json
{
  "cs": "fs",
  "cmd": "tree",
  "payload": {
    "tree": {
      "name": "/",
      "path": "/",
      "ftype": "folder",
      "children": [
        { "name": "src", "path": "/src", "ftype": "folder", "children": [] },
        { "name": "README.md", "path": "/README.md", "ftype": "file" }
      ]
    }
  }
}
```

### Example: `fs/write`

Client → server:

```json
{
  "cs": "fs",
  "cmd": "write",
  "payload": {
    "path": "/src/app.js",
    "changes": [
      {
        "change_type": "insert",
        "change_data": "console.log('hi')\n",
        "start_line": 0,
        "start_char": 0,
        "end_line": 0,
        "end_char": 0
      }
    ]
  }
}
```

Server → caller:

```json
{
  "cs": "fs",
  "cmd": "written",
  "payload": {
    "path": "/src/app.js",
    "revisions": [12]
  }
}
```

Server → other viewers of that file:

```json
{
  "cs": "fs",
  "cmd": "change",
  "payload": {
    "path": "/src/app.js",
    "change_type": "insert",
    "change_data": "console.log('hi')\n",
    "start_line": 0,
    "start_char": 0,
    "end_line": 0,
    "end_char": 0,
    "revision": 12,
    "user_id": 7
  }
}
```

### Example: `chat/message` fanout

Sender:

```json
{ "cs": "chat", "cmd": "message", "payload": { "channel_id": 1, "text": "hello" } }
```

Broadcast to subscribers:

```json
{
  "cs": "chat",
  "cmd": "message",
  "payload": {
    "room_id": "project_42_channel_1",
    "channel_id": 1,
    "user_id": 7,
    "name": "alice",
    "text": "hello",
    "timestamp": "2026-07-15T07:45:00Z"
  }
}
```

These objects are taken from the current worker handlers. The authoritative
source remains the worker implementation.
