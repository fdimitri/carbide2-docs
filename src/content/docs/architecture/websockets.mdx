## Wire format

All WebSocket messages are JSON objects with three top-level fields:

```json
{
  "cs":      "fs",
  "cmd":     "write",
  "payload": { "path": "/src/app.js", "delta": "..." }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `cs` | string | CommandSet â€” routes to the correct handler module |
| `cmd` | string | Command within that commandSet |
| `payload` | object | Command-specific data |

## Server â†’ client messages

Responses and broadcasts use the same format, often with a `ref` field mirroring
the client's request ID:

```json
{
  "cs":      "fs",
  "cmd":     "tree",
  "ref":     "abc123",
  "payload": { "entries": [...] }
}
```

### `terminal`

PTY terminal multiplexing.

In Kubernetes mode, terminal PTYs are executed via a per-project shell sidecar
backend. This keeps terminal toolsets isolated from the server/worker runtime
and constrains filesystem access to the mounted project root.

| Command | Direction | Description |
|---------|-----------|-------------|
| `open` | Câ†’S | Open a new PTY terminal |
| `input` | Câ†’S | Send keystrokes to PTY |
| `output` | Sâ†’C | Broadcast PTY output to subscribers |
| `resize` | Câ†’S | Resize PTY (cols Ă— rows) |
| `close` | Câ†’S | Close a terminal session |

#### Terminal backend modes

| Backend | Execution model | Isolation level |
|---------|------------------|-----------------|
| `local` | `PTY.spawn` directly in workspace pod process space | Lowest (shared runtime namespace) |
| `docker` | `docker exec` into project container | Strong (container boundary) |
| `kube` | `kubectl exec` into project shell sidecar/pod | Strong (pod/container boundary + project-root mount) |

Default production posture is sidecar-backed terminals so tool execution stays
out of the server/worker process namespaces.

## Protocol versioning

The worker advertises two constants on the `system/connected` message:

| Field | Description |
|-------|-------------|
| `protocol` | Current server protocol version (integer) |
| `min_client` | Minimum client version the server accepts |

The client sends `&proto=N&min_server=M` on the WebSocket URL. A mismatch
currently logs a warning (non-blocking); hard-fail will be added in a future release.
