CARB/IDE2 runs as a full Kubernetes stack. Local development uses **k3d**:
`deploy.rb` brings up the cluster, builds every component, and deploys the
server, worker, control plane, and SPA together.

## Prerequisites

| Tool | Minimum version |
|------|----------------|
| Docker | 24.x |
| k3d | 5.x |
| kubectl | 1.28+ |
| Helm | 3.x |
| Ruby | 4.0 (via rbenv) |
| Node.js | 20.x |

## Quick start

```bash
# Clone the meta repo with submodules
git clone --recurse-submodules https://github.com/fdimitri/carbide2.git
cd carbide2

# Build, create the cluster if needed, and deploy everything.
# deploy.rb brings up the k3d cluster + infra itself
# (via carbide2-server/scripts/dev-cluster.sh) before building and deploying.
ruby scripts/deploy.rb
```

See [Kubernetes Ops](/operations/kubernetes/) for cluster management commands.

## Docker Compose (stale — not currently supported)

The committed `carbide2-server/docker-compose.yml` predates the k3d switch. It
brings up a single unified server image against Postgres and has **not** been
kept working since; treat it as unmaintained.

It also can't stand in for the full stack on its own. The server still exchanges
tokens with the **control plane** for JWT auth, and a real deployment hosts many
projects — neither of which a server-only Compose file provides. A working
Compose path would need to run the control plane too. Until then, use k3d.

## Submodule discipline

For local development checkouts, update submodules after any `git pull` on the
meta repo so your working tree matches the pinned SHAs:

```bash
git submodule update --init --recursive
```

(`deploy.rb` does this itself before every deploy — this is only for keeping
your own checkout in sync.)
